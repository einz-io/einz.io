30-Day Launch Checklist (v1 Pilot)

This checklist moves the protocol from conceptual architecture to a functional v1 pilot. It prioritizes the Shield-MCP Wrapper and the Identity Registry as foundational components.

Phase 1: Foundation & Identity (Days 1–10)

Goal: Establish the hardware root of trust and the registration portal.

  • [ ] Day 1: Environment Setup. Configure a reference server with a TPM 2.0 module (or a cloud TEE like AWS Nitro Enclaves).
  • [ ] Day 3: The Identity Provider. Deploy the central Identity Registry (the service that verifies TPM quotes and issues "birth certificates").
  • [ ] Day 5: Merkle Tree Genesis. Generate the Genesis Root Hash, the first version of the global reputation log.
  • [ ] Day 7: Shield-MCP SDK Alpha. Finalize the Python wrapper that automates TPM signing for standard MCP tool calls.
  • [ ] Day 10: PCR Whitelisting. Define the gold-standard hashes for the first supported AI models (e.g., specific versions of Llama 3 or GPT-4o-mini via MCP).

Phase 2: The Pilot Program (Days 11–20)

Goal: Onboard the first trusted agents and test the handshake.

  • [ ] Day 11: Developer Portal. Launch a simple site where developers can download the shield-mcp SDK and read documentation.
  • [ ] Day 13: The First 10 Agents. Recruit 10 pilot developers to register their hardware and agents.
  • [ ] Day 15: Peer-to-Peer Handshake Test. Execute a task where Agent A verifies Agent B’s Merkle proof locally before delegating.
  • [ ] Day 18: Simulated Attack. Attempt to register a spoofed (software-only) agent to confirm TPM verification correctly rejects it.
  • [ ] Day 20: Performance Logging. Ensure every pilot task generates a valid TPM-signed evidence packet.

Phase 3: Judiciary & Scale (Days 21–30)

Goal: Deploy the automated auditing layer and publish the public root.

  • [ ] Day 21: Judge Agent Deployment. Deploy the first 3 system Judge Agents (platform-run) to audit logs.
  • [ ] Day 24: First Reputation Update. Perform the first batch update of the Merkle tree based on pilot outcomes.
  • [ ] Day 27: Revocation List Test. Test the kill switch by manually revoking one pilot agent and verifying handshake denial.
  • [ ] Day 29: Transparency Dashboard. Launch a public site showing the current global root hash and number of verified hardware modules.
  • [ ] Day 30: Public Alpha Launch. Open the platform for general developer registration.

Critical Success Metrics (Day 30)

  • Handshake Latency: Direct P2P verification is under 200 ms.
  • Hardware Integrity: 100% of active agents have a verified TPM EK.
  • Audit Coverage: 100% of tasks have a signed evidence packet stored in the transparency log.
